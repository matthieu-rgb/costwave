# Security & Feature Backlog

## Phase 3 Security Audit - Low Priority Findings

### [LOW] #1: Validation Cost Not Rate-Limited Per User

**Context:** The `addProvider` action validates API keys by making actual API calls to provider endpoints. No rate limiting on how many times a user can attempt to add a provider.

**Risk:** Compromised account could be used as API key validation oracle. Attacker could rapidly validate stolen API keys against multiple providers.

**Recommendation:** Add rate limiting (max 10 provider additions per hour per user) using pg-boss or similar. Project CLAUDE.md already mandates rate limiting on signup/login/ingest - extend to provider operations.

**Priority:** Low (requires authenticated session)

---

### [LOW] #2: SSE Polling Interval May Allow Stale Session Attacks

**Context:** SSE endpoint `/api/stream/[userId]/route.ts` verifies session once at connection time but continues streaming events every 2 seconds without re-validation. If user's session is revoked (logout, password change, admin action), SSE connection continues until client disconnects.

**Risk:** Events being streamed are non-sensitive (usage data, not API keys). However, this violates immediate session invalidation principle.

**Recommendation:** Periodically re-validate session (every 60 seconds) or implement session change notifications via Better Auth's session revocation hooks.

**Priority:** Low (non-sensitive data streaming)

---

### [LOW] #3: Vault Key Derivation Salt Uses Only 16 Bytes

**Context:** Salt for Argon2id is generated by taking first 32 hex characters (16 bytes) of SHA-256 hash in `/Users/matthieu/Dev/costwave/app/src/lib/crypto/vault.ts:26-30`.

**Risk:** None - 16 bytes (128 bits) is adequate for salt uniqueness. This is best practice observation.

**Recommendation:** Use full 32-byte SHA-256 output:
```typescript
const salt = crypto
  .createHash('sha256')
  .update(`${userId}:${pepper}`)
  .digest('hex'); // Full 64 hex chars = 32 bytes
```

**Priority:** Low (optional improvement)

## Phase 11.5 - Refactor provider validation (V1.5)

**Problem**: Hardcoded model names in validateKey() can become deprecated, causing provider addition failures.

**Root cause**: Each provider's validateKey() makes POST /v1/messages (or equivalent) with a hardcoded model name to test API key validity. When providers deprecate models, validation breaks.

**Solution**: Refactor validateKey() to use /v1/models endpoint instead of /v1/messages for all providers that support it. This avoids dependency on specific model availability.

**Alternative approach**: Keep current implementation but use most stable "latest" model aliases (e.g., mistral-small-latest) where available.

**Estimation**: 2-3h

**Priority**: Medium - only fix when another model gets deprecated
